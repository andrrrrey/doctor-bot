## 1. Что это за проект и какую задачу он решает

**Doctor Bot** — это медицинская «воронка» для невролога. Пациент с болью в спине проходит интерактивный опросник, система автоматически прикидывает наиболее вероятную причину боли (мышцы, грыжа, артроз, стеноз, воспаление, психосоматика) и предлагает записаться на консультацию. Заявка пациента сразу улетает врачу на почту (с PDF-отчётом) и в CRM Bitrix24.

Главная ценность проекта — **не код, а заложенный в него врачебный опыт**. Алгоритм определения диагноза — это формализованные 17 лет клинической практики доктора. По сути это «экспертная система»: набор правил вида «если пациенту больше 60 лет — повысь вероятность артроза и стеноза, понизь вероятность мышечной боли». Поэтому ключевые архитектурные решения крутятся вокруг одного вопроса: **как сохранить и не потерять эту врачебную логику, при этом дав возможность её менять и отслеживать изменения**.

## 2. Общая картина: три независимых части

Проект намеренно разбит на три самостоятельных приложения, которые общаются только через HTTP (REST API):

```
┌─────────────────────────────────────────────────────────────┐
│                                                               │
│   КЛИЕНТЫ (кто задаёт вопросы пациенту)                       │
│                                                               │
│   ┌──────────┐   ┌──────────┐   ┌────────────────┐           │
│   │  Сайт /  │   │  Виджет  │   │ Telegram-бот   │           │
│   │  client  │   │ (public) │   │ (отдельный     │           │
│   │          │   │          │   │  процесс)      │           │
│   └────┬─────┘   └────┬─────┘   └───────┬────────┘           │
│        │              │                 │                     │
│        └──────────────┼─────────────────┘                    │
│                       │ HTTP / REST                          │
│                       ▼                                       │
│            ┌─────────────────────┐                           │
│            │   СЕРВЕР (Express)   │  ← вся бизнес-логика      │
│            │   + алгоритм         │                           │
│            │   + интеграции       │                           │
│            └──────────┬──────────┘                           │
│                       │                                       │
│        ┌──────────────┼──────────────┐                       │
│        ▼              ▼              ▼                        │
│   ┌─────────┐   ┌──────────┐   ┌──────────┐                  │
│   │PostgreSQL│  │  Email   │   │ Bitrix24 │                  │
│   │ (Prisma) │  │  (SMTP)  │   │  (CRM)   │                  │
│   └─────────┘   └──────────┘   └──────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

**Почему именно так, а не один монолит?**

- **Разные каналы привлечения — разные клиенты.** Опросник нужно показывать и на своём сайте, и встроенным виджетом на чужих сайтах-партнёрах, и в Telegram. У всех трёх каналов разный «способ показать вопрос» (HTML-форма против сообщений с кнопками в мессенджере), но **одинаковая логика диагностики**. Логично вынести логику на сервер, а клиентов сделать «тонкими».
- **Telegram-бот — отдельный процесс.** Он работает в режиме постоянного опроса серверов Telegram (long polling) и должен жить своей жизнью независимо от веб-сервера. Если упадёт бот — сайт продолжит работать, и наоборот. Их даже можно запускать порознь (для этого есть два разных PM2-конфига).
- **Сервер — единственный источник правды.** Алгоритм диагноза, доступ к базе, отправка писем и лидов — всё в одном месте. Клиенты не считают диагноз сами (за одним историческим исключением, см. раздел 7), они только собирают ответы и показывают результат.

## 3. Поток данных: путь пациента

Разберём на примере виджета (самый современный путь), что происходит шаг за шагом:

1. **Старт.** Пациент открывает виджет. Фронтенд создаёт «сессию» — `POST /api/sessions`. Сессия нужна, чтобы потом посчитать воронку: сколько людей начали, сколько дошли до конца. Это анонимная метка, без персональных данных.
2. **Загрузка вопросов.** Фронтенд запрашивает `GET /api/questions` — список активных вопросов из базы, уже сгруппированных по принципу «основной вопрос + его уточняющие подвопросы».
3. **Прохождение.** Вопросы показываются по одному. Ответ на предыдущий вопрос определяет, какие подвопросы покажутся дальше (условная логика). Всё это происходит в браузере.
4. **Расчёт диагноза.** Собранные ответы отправляются на сервер, тот прогоняет их через алгоритм и возвращает готовый HTML с результатом и рекомендациями.
5. **Завершение.** Сессия помечается завершённой: `PATCH /api/sessions/:id/complete`.
6. **Заявка.** Если пациент захотел консультацию — он оставляет имя, телефон, город. Это `POST /api/submissions`. Сервер:
   - сохраняет заявку в базу;
   - **сразу отвечает клиенту** «принято» (HTTP 201);
   - а уже **после ответа**, в фоне, генерирует PDF, шлёт письмо врачу и создаёт лид в Bitrix24.
7. **Файл (опционально).** Пациент может приложить архив со снимками МРТ — `POST /api/submissions/:id/file`.

> **Важное архитектурное решение — «ответь сначала, отправь потом» (fire-and-forget).** Отправка письма и обращение к Bitrix могут быть медленными или временно отвалиться. Заставлять пациента ждать, пока письмо дойдёт, — плохо: он закроет вкладку, подумав, что всё зависло. Поэтому сервер сначала сохраняет заявку и отвечает «ок», а почту и CRM дёргает в фоне. Если Bitrix недоступен — заявка **всё равно не потеряна**, она лежит в базе со статусом `error`, и её можно переотправить вручную из админки. Это типичный паттерн для надёжной работы с внешними сервисами, которые тебе неподконтрольны.

## 4. Сервер (backend)

**Стек:** Node.js + Express 4 + TypeScript, ORM Prisma 7, база PostgreSQL.

### Почему Express, а не что-то модное

Express — самый простой и проверенный веб-фреймворк для Node. Проект по объёму небольшой, экзотических требований нет, команда, скорее всего, маленькая. Брать тяжёлый фреймворк (NestJS и т.п.) здесь было бы избыточно. Express даёт ровно столько, сколько нужно: маршруты, middleware, отдача статики.

### Как организован код

Точка входа — `server/src/server.ts`. Здесь:
- подключается отдача статических файлов (виджет, админка, загруженные файлы);
- настраивается CORS (открыт для всех — об этом ниже);
- регистрируются все маршруты.

Маршруты разнесены по папкам по смыслу:

```
routes/
├── questions.ts      # публичные: вопросы для виджета
├── sessions.ts       # публичные: сессии (для воронки)
├── submissions.ts    # публичные: приём заявок + загрузка файлов
├── settings.ts       # публичные настройки (тексты, тема)
└── admin/            # всё, что требует входа администратора
    ├── auth.ts          # вход, смена пароля
    ├── users.ts         # управление администраторами
    ├── questions.ts     # редактор вопросов
    ├── submissions.ts   # просмотр заявок
    ├── stats.ts         # аналитика/воронка
    ├── settings.ts      # SMTP, Bitrix
    └── versions.ts      # версии опросника
```

Бизнес-логика, которую переиспользуют несколько маршрутов, вынесена в отдельные модули верхнего уровня:

- `process-answers.ts` — **алгоритм диагностики** (главный модуль, см. раздел 5);
- `generate-pdf.ts` — сборка PDF-отчёта из ответов (через PDFKit, со встроенным шрифтом DejaVuSans для поддержки кириллицы);
- `transporter.ts` — отправка почты через Nodemailer;
- `bitrix.ts` — создание лида в Bitrix24;
- `questionnaire-version.ts` — снимки версий опросника;
- `db.ts` — единственный экземпляр Prisma-клиента (чтобы не плодить подключения к базе);
- `constants.ts` — тексты диагнозов, рекомендации и «старый» список вопросов.

**Почему так:** разделение «маршрут отдельно, логика отдельно» позволяет переиспользовать один и тот же код. Например, отправка письма и лида нужна и в новом маршруте `/api/submissions`, и в старом `/postConsultationData` — поэтому она вынесена в `transporter.ts` и `bitrix.ts`, а не скопирована дважды.

### Prisma 7 с драйвер-адаптером

Используется относительно свежая Prisma 7 с пакетом `@prisma/adapter-pg`. Это новый способ подключения, где Prisma работает поверх обычного драйвера `pg`. Плюс — больше контроля над пулом соединений и совместимость с серверless/edge-средами. Для проекта это «работа на вырост»: миграции описаны в `prisma/migrations`, схема — в `schema.prisma`, типы генерируются автоматически, так что весь доступ к базе типобезопасный.

## 5. Алгоритм диагностики — сердце проекта

Это самая важная и самая нестандартная часть. Находится в `server/src/process-answers.ts`. Разберём подробно, потому что именно здесь живёт врачебная экспертиза.

### Идея: баллы по шести «осям»

Система не пытается выдать один диагноз. Вместо этого она ведёт **счётчик баллов по шести направлениям** одновременно:

```ts
{
  neurosis: 0,      // психосоматика, невроз
  muscles: 0,       // мышечная боль
  hernia: 0,        // грыжа межпозвонкового диска
  arthrosis: 0,     // артроз
  stenosis: 0,      // стеноз позвоночного канала
  inflammation: 0,  // воспаление
}
```

Каждый ответ пациента **прибавляет или отнимает баллы** у одной или нескольких осей. Например:

- Возраст до 25 лет → `+2` к мышцам (у молодых чаще мышечная боль).
- Возраст 60+ → `+3` артроз, `+2` стеноз, `−1` грыжа, `−3` мышцы (с возрастом меняется типичная причина).
- Боль отдаёт в одну ногу → `+3` грыжа (классический признак защемления корешка).
- Не может пройти определённую дистанцию → `+4` стеноз.

В конце система смотрит, какие оси набрали больше всего, и так формирует итоговый диагноз.

### Почему «правила в коде», а не нейросеть/ML

Это сознательный и правильный для медицины выбор:

1. **Объяснимость.** Каждый балл можно проследить: «почему система решила, что это грыжа? — потому что боль в одну ногу (+3), возраст 30 (+2)…». Нейросеть так не умеет, а в медицине прозрачность критична.
2. **Нет данных для обучения.** Чтобы обучить ML-модель, нужны тысячи размеченных историй болезни. Их нет. А врачебная эвристика — есть, прямо в голове у доктора. Её проще «закодировать», чем собирать датасет.
3. **Полный контроль врача.** Доктор может сказать «прибавь сюда два балла» — и разработчик ровно так и сделает. Поведение предсказуемо.

### Многопроходная обработка и поправка на психосоматику

Самое хитрое место — функция `secondProcessAnswers`. Алгоритм считает диагноз **в два прохода**, и второй проход зависит от уровня невроза, выявленного на первом:

- Если **невроза нет вообще** (`neurosis === 0`) — система считает, что пациент, наоборот, **недооценивает** свою боль, и искусственно поднимает его оценку боли на 2 пункта, после чего пересчитывает всё заново.
- Если невроз **умеренный** (4–7 баллов) — он «съедает» часть симптомов: алгоритм немного понижает баллы грыжи/артроза/стеноза и поднимает мышцы, потому что при неврозе человек склонен преувеличивать. Боль реальна, но её органическая причина мягче, чем кажется.
- Если невроз **сильный** (>7 баллов) — наоборот, оценку боли снижают на 2, дают базовый бонус мышцам и пересчитывают.

> **Зачем такая сложность?** Это и есть «соль» врачебного опыта: один и тот же набор симптомов у тревожного и у спокойного пациента означает разные вещи. Тревожный преувеличит, стойкий — преуменьшит. Алгоритм пытается это скомпенсировать. Формализовать такое в простой формуле невозможно — отсюда условные ветки и пересчёты.

### Выбор финального диагноза

Функция `processResultDiagnosis` берёт четыре «органические» оси (мышцы, грыжа, артроз, стеноз), отбрасывает самую слабую, считает среднее по остальным и оставляет только те, что **выше среднего**. Плюс набор «специальных случаев» — например, если в лидерах одновременно грыжа и артроз, побеждает грыжа; особая обработка стеноза, который редко ходит в одиночку. Это снова не математика ради математики, а закодированные клинические наблюдения.

### Гибрид: жёсткие правила + настраиваемые веса

Важный нюанс. Помимо жёстко зашитых в код правил, есть **второй, настраиваемый слой**. У каждого варианта ответа в базе есть поле `weights` — словарь вида `{ "hernia": 2, "muscles": -1 }`. Когда приходят ответы, сервер дополнительно прибавляет эти веса к диагнозу:

```ts
// из process-answers.ts
if (optionWeights) {
  answers.forEach(({ questionId, questionAnswers }) => {
    questionAnswers.forEach((value) => {
      const w = optionWeights.get(`${questionId}::${value}`);
      if (w) { diagnosis.hernia += w.hernia; /* ...и т.д. */ }
    });
  });
}
```

**Зачем два слоя сразу?** Жёсткие правила в коде — это «ядро», сложная логика с зависимостями между ответами (её через UI не выразить). А веса в базе — это «ручки настройки» для врача: он может через админку, **без программиста**, подкрутить влияние конкретного ответа на конкретный диагноз. Получается баланс между гибкостью и сохранением сложной логики.

## 6. База данных и почему такая схема

Схема описана в `server/prisma/schema.prisma`. Разберём таблицы и заложенные в них идеи.

| Таблица | Назначение | Ключевая идея |
|---|---|---|
| `Admin` | Администраторы | Роль (`superadmin`/`doctor`), пароль хранится только в виде bcrypt-хеша |
| `Question` | Вопросы опросника | Самоссылка `parentId` + `conditions` для уточняющих вопросов |
| `AnswerOption` | Варианты ответов | Поле `weights` (JSON) — те самые настраиваемые веса для алгоритма |
| `Session` | Анонимная сессия прохождения | Нужна для воронки конверсии, без персональных данных |
| `Submission` | Заявка пациента | Хранит и данные пациента, и **снимок ответов**, и статус отправки в Bitrix |
| `QuestionnaireVersion` | Версия опросника | Снимок всех вопросов + хеш для дедупликации |
| `Setting` | Настройки (ключ-значение) | SMTP, Bitrix и пр. — редактируются на лету через админку |

Несколько решений заслуживают объяснения:

**1. Вопросы хранятся деревом через `parentId` и `conditions`.**
Вопрос может быть «уточняющим»: он показывается, только если на родительский вопрос дан определённый ответ. Вместо отдельной таблицы связей это решено самой простой моделью — у дочернего вопроса есть ссылка на родителя (`parentId`) и список условий (`conditions`), при которых он появляется. Маршрут `GET /api/questions` собирает из плоского списка удобное дерево «вопрос → подвопросы».

**2. В `Submission.answers` лежит JSON, а не связи на таблицы.**
Ответы пациента сохраняются как готовый JSON-снимок («вопрос: ответы»), а не ссылками на таблицу вопросов. Почему так? Потому что **вопросы со временем меняются** — текст правят, варианты удаляют. Если бы заявка ссылалась на текущие вопросы, то старая заявка «поехала» бы после любой правки. А JSON-снимок навсегда фиксирует, **что именно** ответил пациент тогда. Для медицинских данных это обязательное требование: история не должна меняться задним числом.

**3. Настройки (SMTP, Bitrix) — в базе, а не только в `.env`.**
Логин почты, пароль, список получателей, webhook Bitrix лежат в таблице `Setting` и редактируются через UI. Переменные окружения служат запасным значением. **Зачем?** Чтобы врач/администратор мог поменять почту или адрес CRM **сам, через браузер**, без доступа к серверу и без перезапуска. Это удобно для нетехнического заказчика.

**4. Статус отправки в Bitrix хранится прямо в заявке** (`bitrixStatus`, `bitrixError`).
Это и есть страховка для упомянутого fire-and-forget: видно, какие заявки ушли в CRM, какие упали и почему, и любую можно переотправить.

## 7. Две системы вопросов: «старая» и «новая»

Это, пожалуй, главная особенность кодовой базы, которую нужно понимать. В проекте **сосуществуют два способа** задавать вопросы — и это след эволюции проекта.

### «Старая» система (legacy)

- Вопросы **жёстко прописаны в коде** — массив `QUESTIONS` в `server/src/constants.ts`.
- Маршруты: `GET /survey` (отдаёт этот массив), `POST /postTestResults` (считает диагноз), `POST /postConsultationData` (приём заявки).
- Кто использует: **Telegram-бот** и, исторически, исходный сайт в папке `client/`.

### «Новая» система

- Вопросы **хранятся в базе** и редактируются через админку.
- Маршруты: `GET /api/questions`, `POST /api/sessions`, `POST /api/submissions`.
- Появились сессии (воронка), версионирование, загрузка файлов, статусы Bitrix.
- Кто использует: **виджет** (`public/`) и админка.

### Почему так вышло и почему обе живут

Проект явно начинался как простой захардкоженный опросник (папка `client/` + константы на сервере), под него написали Telegram-бота. Потом понадобилось редактировать вопросы без программиста, считать конверсию и хранить заявки — так родилась «новая» система с базой и админкой.

Старые маршруты **намеренно оставлены** (в `server.ts` они прямо помечены `// Legacy routes (kept for backward compatibility with Telegram bot)`), потому что Telegram-бот пока работает через них. Переписывать бота на новый API — отдельная задача, которую отложили. Это нормальный, осознанный технический компромисс: «не ломай то, что работает, пока не дошли руки переписать».

> **Что это значит на практике:** если врач меняет вопросы в админке — это видит **виджет**, но **не Telegram-бот** (у него вопросы зашиты в код). Это важно держать в голове при сопровождении. Полное объединение на единый API через базу — очевидный следующий шаг развития.

## 8. Фронтенд: сайт, виджет и админка

Весь фронтенд написан на **чистом JavaScript (Vanilla JS)** без фреймворков, со стилями на основе лёгкой библиотеки **Pico CSS** (классы почти не нужны, оформление «из коробки»).

**Почему без React/Vue?** Задача фронтенда простая: показать вопросы по очереди и собрать ответы. Тащить ради этого сборщик, npm-зависимости и фреймворк — лишняя сложность и вес. Чистый JS грузится мгновенно, что особенно важно для **встраиваемого виджета** на чужих сайтах: он должен быть лёгким и ни с чем не конфликтовать.

Фронтенд разбит на маленькие модули по ответственности (ES-модули, `import`/`export`):

- `questions.js` — отрисовка вопросов и условная логика «показать следующий»;
- `collect-answers.js` — сбор ответов из формы;
- `validation.js` — проверка заполнения;
- `consultation.js` — форма записи на консультацию;
- `script.js` / `widget.js` — точка входа, склейка всего вместе.

Есть фактически три варианта интерфейса:
- **`client/`** — исходный standalone-сайт (легаси, считает часть диагноза на клиенте, работает через старые маршруты);
- **`public/widget.*`** — встраиваемый виджет (новый API, с согласием на обработку данных);
- **`public/widget-steps.*`** — пошаговый вариант виджета (вопросы по одному).

### Админка

`public/admin/` — отдельное одностраничное приложение (тоже Vanilla JS, ~1500 строк в `app.js`). Общается с сервером через `/api/admin/*`, токен хранит у себя и подставляет в заголовок `Authorization`. Через неё врач управляет вопросами, весами, заявками, смотрит воронку и версии. Сервер отдаёт её как обычную статику по адресу `/admin`.

## 9. Telegram-бот

Отдельное приложение в `telegram/` (TypeScript), запускается своим процессом.

**Режим работы — long polling, а не webhook.** Бот сам постоянно спрашивает у серверов Telegram «есть новые сообщения?» (`index.ts`, опции `polling`). Почему polling, а не webhook? Polling **не требует публичного HTTPS-адреса и SSL-сертификата** — бот можно запустить где угодно, хоть за NAT. Это проще в развёртывании. Webhook эффективнее под высокой нагрузкой, но для потока «несколько пациентов в день» это неважно. В коде, кстати, оставлен и обработчик ошибок webhook — на случай, если в будущем захотят переключиться.

**Состояние диалога хранится в памяти процесса** — `const userStates: UserStates = new Map()`. Для каждого пользователя бот помнит, на каком он вопросе и что уже ответил. **Компромисс:** если бот перезапустить, все незавершённые диалоги теряются. Но это приемлемо: пациент просто начнёт заново по `/start`, а тащить ради этого внешнее хранилище (Redis и т.п.) — избыточно для текущих объёмов.

Код бота аккуратно разложен по обработчикам:
- `handlers/commands.ts` — команды вроде `/start`;
- `handlers/messageHandlers.ts` — текстовые сообщения (например, ввод возраста);
- `handlers/callbackHandlers.ts` — нажатия на inline-кнопки (варианты ответов);
- `api.ts` — общение с backend (берёт вопросы из `/survey`, шлёт ответы в `/postTestResults`).

Важно: бот ходит на **старые** маршруты сервера (см. раздел 7), а адрес сервера переключается между localhost и боевым доменом по `NODE_ENV`.

## 10. Интеграции: Email и Bitrix24

### Email (Nodemailer + PDF)

После заявки врачу уходит письмо с данными пациента и **PDF-вложением**, где собраны все ответы опросника. PDF собирается на лету через PDFKit (`generate-pdf.ts`). Отдельно подключён шрифт `DejaVuSans.ttf` — потому что стандартные шрифты PDFKit не умеют кириллицу, и без него русский текст в отчёте превратился бы в «кракозябры». Настройки SMTP берутся из базы (с откатом на `.env`).

### Bitrix24 (CRM)

`bitrix.ts` создаёт лид через REST-вебхук Bitrix (метод `crm.lead.add`). Адрес вебхука и заголовок лида берутся из настроек в базе. В комментарий лида складывается диагноз, расширенный диагноз и все ответы — чтобы менеджер видел контекст ещё до звонка.

**Обработка ошибок продумана:** если webhook не задан — понятная ошибка; если Bitrix вернул HTTP-ошибку или ошибку API — она пробрасывается наверх и **сохраняется в заявку** (`bitrixStatus: 'error'`, текст в `bitrixError`). Благодаря этому ни один лид не пропадает молча — всё видно в админке и переотправляется вручную.

### Аналитика

В HTML-страницы встроены счётчики **Яндекс.Метрики** и **Roistat** для сквозной аналитики и отслеживания событий воронки на стороне маркетинга.

## 11. Безопасность и авторизация

**Аутентификация админки — JWT + bcrypt** (`middleware/adminAuth.ts`).

- Пароли админов хранятся **только в виде bcrypt-хеша**, в открытом виде их нет нигде.
- При входе сервер выдаёт JWT-токен на **24 часа**. Дальше клиент шлёт его в заголовке `Authorization: Bearer ...`, а middleware `adminAuth` проверяет подпись на каждом защищённом запросе.
- **Две роли** разделены на уровне middleware: `requireSuperadmin` пускает к опасным разделам (вопросы, пользователи, настройки) только суперадмина, а `doctor` может лишь смотреть заявки и аналитику. Принцип наименьших привилегий: врачу не нужен доступ к настройкам SMTP, и он его не получает.
- Защиты от «выстрела себе в ногу»: нельзя удалить последнего суперадмина и нельзя понизить в правах самого себя — иначе можно случайно потерять доступ ко всей системе.

**CORS открыт для всех** (`Access-Control-Allow-Origin: *`). Это **намеренно**: виджет должен работать, будучи встроенным на любой партнёрский сайт с произвольным доменом. Платой за это является открытость публичных эндпоинтов — но они и так публичные (вопросы, приём заявок), чувствительных данных через них не утекает, а вся админка закрыта токеном.

**Загрузка файлов строго ограничена** (`routes/submissions.ts`):
- разрешены только архивы и снимки — `.zip`, `.rar`, `.dcm` (DICOM);
- максимум 50 МБ (защита от переполнения диска);
- **минимум 20 МБ** — необычное правило: слишком маленький файл считается «битым/неполным» снимком МРТ и отклоняется с понятным сообщением. Это доменная валидация: настоящий архив МРТ просто не бывает крошечным.

> Что стоит улучшить: `JWT_SECRET` имеет небезопасное значение по умолчанию — на проде его обязательно нужно задать через `.env`. На это указано и в README.

## 12. Версионирование опросника

Очень продуманная для медицинского продукта вещь (`questionnaire-version.ts`).

**Проблема:** опросник со временем меняется — добавляют вопросы, правят веса. Если через полгода посмотреть на старую заявку, как понять, **по какой именно версии алгоритма** её посчитали?

**Решение:** при каждой заявке система делает «снимок» (snapshot) текущего состояния всех вопросов и их весов, считает от него **SHA-256 хеш** и сохраняет как версию. Заявка ссылается на конкретную версию.

```ts
// упрощённо из questionnaire-version.ts
const snapshot = await buildQuestionnaireSnapshot();
const hash = hashSnapshot(snapshot);
const existing = await prisma.questionnaireVersion.findUnique({ where: { hash } });
if (existing) return existing.id;   // версия не изменилась — переиспользуем
// иначе создаём новую версию с инкрементом номера
```

**Изящность в дедупликации через хеш:** новая запись версии создаётся, **только если содержимое реально изменилось**. Пока опросник не трогают, все заявки указывают на одну и ту же версию. Изменили хоть один вес — хеш поменялся, появилась версия №2, и новые заявки идут к ней. Так получается точная история без дубликатов: видно, по какому набору вопросов посчитана каждая заявка, и сколько заявок собрала каждая версия.

## 13. Развёртывание и эксплуатация

**Процесс-менеджер — PM2.** В корне два конфига:
- `start.config.js` — поднимает сервер и Telegram-бот вместе;
- `start-telegram.config.js` — только бот (чтобы можно было крутить их на разных машинах/перезапускать порознь).

PM2 берёт на себя автоперезапуск при падении, логи и автозапуск при перезагрузке сервера. Это разумный минимум для небольшого Node-проекта — без Docker/Kubernetes, которые здесь были бы оверкиллом.

**Сборка:** и сервер, и бот пишутся на TypeScript и компилируются в JS (`tsc`) в папку `dist/`, которая и запускается на проде. В разработке — `nodemon` + `ts-node` с горячей перезагрузкой.

**Миграции базы** применяются через Prisma (`prisma migrate deploy`), начальные данные (вопросы и первый суперадмин) заливаются скриптом `prisma/seed.ts`.

**Файлы пациентов** складываются в папку `uploads/` и отдаются сервером как статика по `/uploads`. (На вырост стоит подумать про вынос в облачное хранилище, чтобы не привязываться к диску конкретной машины.)

## 14. Сильные стороны, слабые места и технический долг

Честная оценка, чтобы было понятно, куда развивать проект.

### Что сделано хорошо

- **Чёткое разделение клиентов и сервера.** Логика в одном месте, клиенты тонкие.
- **Надёжный приём заявок.** Заявка сохраняется до отправки во внешние сервисы и не теряется при их сбоях; есть статусы и ручная переотправка.
- **Версионирование с дедупликацией по хешу** — зрелое решение, важное для медицины и аудита.
- **Неизменяемые снимки ответов** в заявках — история не «едет» при правках вопросов.
- **Гибридный алгоритм** — сложная логика в коде плюс настраиваемые врачом веса в базе.
- **Прагматичный выбор технологий** — никакого оверинжиниринга: Express, Vanilla JS, PM2 ровно под размер задачи.

### Слабые места и долг

- **Дублирование двух систем вопросов** (раздел 7). Telegram-бот живёт на старом захардкоженном списке и не видит правок из админки. Главный кандидат на рефакторинг — перевести бота на `/api/questions`.
- **Состояние Telegram-бота только в памяти** — теряется при перезапуске.
- **`JWT_SECRET` с дефолтом** — потенциальная дыра, если забыть переопределить на проде.
- **Алгоритм диагностики не покрыт автотестами** (их в проекте, похоже, нет вовсе). Для такой ответственной и сложной логики тесты были бы очень кстати: любая правка весов может незаметно сломать другой сценарий.
- **Файлы хранятся на локальном диске** — мешает горизонтальному масштабированию.
- **В репозитории лежит скомпилированный `dist/`** — обычно его держат вне гита и собирают при деплое.

### Куда расти дальше (логичные следующие шаги)

1. Перевести Telegram-бот на новый API из базы — убрать дублирование.
2. Покрыть алгоритм диагностики тестами (это снизит страх перед правками весов).
3. Вынести загрузки в облачное хранилище.
4. Усилить базовую безопасность (обязательный секрет, лимиты на публичные эндпоинты).
